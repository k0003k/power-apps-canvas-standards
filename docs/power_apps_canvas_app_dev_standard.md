# Power Apps キャンバスアプリ開発標準

作成日：2026-06-24  
改訂：2026-06-24（コントロール名のアプリ全体一意制約を反映）

## 0. 目的・適用範囲

本標準は、Power Apps キャンバスアプリを、保守しやすく、読みやすく、実装者間で迷いにくい形で作成するための最低限の開発標準である。

人間の開発者だけでなく、AIによるレビュー、改修、リファクタリング時にも、画面構造、命名、処理意図を理解しやすい状態を目指す。

前提は以下とする。

- モダンコントロールを主に使用する
- SharePoint リストを主要データソースとして使用する
- 固定の `X` / `Y` 指定は極力使用しない
- コンテナベースでレイアウトを構成する
- レスポンシブ対応は必須としない
- 独自定義要素は snake_case を基本とする
- 標準関数は Power Fx 標準の CamelCase に従う

本標準から外れる場合は、理由を説明できることを条件とする。

## 1. 命名規則

### 1.1 基本方針

命名は、検索性、参照性、可読性を重視する。

命名では以下を意識する。

- 何の部品か分かること
- どの役割か分かること
- 長すぎないこと
- アプリ全体で一意であること
- 画面ツリーや数式内で探しやすいこと
- 標準関数と独自定義要素を見分けやすいこと

独自定義要素は snake_case を基本とする。

```text
例：
search_button
capture_result_screen
body_container
sp_license_application
```

ただし、コントロールを Power Fx の数式から参照する場合は、種別プレフィックスを付与する。

```text
例：
btn_submit
txt_comment
ico_back
gal_requests
```

### 1.2 アプリ全体での一意性

Power Apps のコントロール名は、スクリーン単位ではなくアプリ全体で一意にする。

そのため、`header`、`body`、`footer` のような画面をまたいで重複しやすい名前は使用しない。  
画面固有の部品には、原則として画面または機能を表すサフィックスを付与する。

```text
例：
header_capture
body_capture
footer_capture
action_area_capture
capture_guide_card_capture
```

画面サフィックスは、`scr_` を除いた画面名を使用する。

```text
例：
scr_capture        -> _capture
scr_search_result  -> _search_result
```

数式から参照するコントロールは、種別プレフィックスと画面サフィックスを組み合わせる。

```text
例：
btn_submit_capture
lbl_title_capture
ico_back_capture
gal_requests_search_result
frm_request_edit_capture
```

### 1.3 プレフィックス省略時の扱い

数式から参照しない小規模コントロールは、命名変更を必須としない。  
その場合は、Power Apps のデフォルト名を許容する。

```text
例：
Button1
Label3
Icon2
```

ただし、独自に名前を付ける場合は、原則として種別プレフィックスまたは画面サフィックスを付与し、アプリ全体で一意にする。

良い例：

```text
btn_submit_capture
lbl_title_capture
ico_back_capture
header_capture
```

避ける例：

```text
submit
title
back
comment
header
body
footer
```

### 1.4 必須プレフィックス

以下の要素は、種別の判別性が重要なため、プレフィックスを必須とする。

| 種別 | プレフィックス | 例 |
|---|---|---|
| Screen | `scr_` | `scr_home` |
| Gallery | `gal_` | `gal_requests` |
| Form | `frm_` | `frm_request_edit_capture` |
| Component | `cmp_` | `cmp_common_header` |
| User Defined Function | `fn_` | `fn_validate_request()` |

### 1.5 任意プレフィックス

以下のコントロールは、Power Fx から参照する場合にプレフィックスを付与する。

| 種別 | プレフィックス | 例 |
|---|---|---|
| Button | `btn_` | `btn_submit` |
| Label | `lbl_` | `lbl_title` |
| Text input | `txt_` | `txt_comment` |
| Combo box | `cmb_` | `cmb_approver` |
| Dropdown | `drp_` | `drp_status` |
| Icon | `ico_` | `ico_back` |
| Image | `img_` | `img_license` |
| Group | `grp_` | `grp_action_buttons` |

### 1.6 画面名

画面名は、画面の業務上の役割を表す。  
`Screen` は `scr_` プレフィックスを必須とする。

```text
例：
scr_home
scr_capture
scr_capture_result
scr_search_result
```

### 1.7 コンテナ名

コンテナは、見た目ではなくレイアウト上の役割で命名する。

コンテナは画面をまたいで同じ役割名が発生しやすいため、原則として画面または機能を表すサフィックスを付与する。

```text
例：
root_capture
header_capture
body_capture
footer_capture
content_capture
search_area_search
filter_area_search
action_area_capture
left_panel_search_result
right_panel_search_result
```

コンテナはコードから参照する場合であっても、原則として `con_` プレフィックスは付与しない。

垂直コンテナ、水平コンテナ、通常コンテナの種類は、画面ツリー上のアイコンで判別できるため、原則として `vertical` / `horizontal` のサフィックスは付与しない。

ただし、同一画面内で同じ役割のコンテナが複数あり、方向を名前に含めないと判別しづらい場合のみ、補助的に方向名を使用してよい。

```text
例：
action_top_capture
action_bottom_capture
```

### 1.8 変数名

グローバル変数は接頭語なしで命名する。

```text
例：
login_user
selected_request
user_role
```

ローカル変数はアンダースコア始まりで命名する。

```text
例：
_is_loading
_is_edit_mode
_error_message
```

### 1.9 コレクション名

コレクションは複数形で命名する。

```text
例：
requests
approvers
search_results
validation_errors
```

### 1.10 データソース名

データソースは、種別を判別できる接頭辞を付与する。

| 種別 | 接頭語 | 例 |
|---|---|---|
| SharePoint | `sp_` | `sp_requests` |
| Dataverse | `dv_` | `dv_accounts` |
| API | `api_` | `api_weather` |

### 1.11 Named Formula

Named Formula はプレフィックスを付与しない。

```text
例：
login_user
is_admin
current_fiscal_year
base_screen_width
```

Named Formula は、読み取り専用の計算値、参照値、共通定数として使用する。  
Named Formula を、グローバル変数経由で関数のように使うことはしない。

### 1.12 User Defined Function

処理を再利用する場合は、User Defined Function の利用を強く推奨する。

User Defined Function は `fn_` プレフィックスを付与し、処理内容が分かる動詞句で命名する。

```text
例：
fn_get_login_user()
fn_validate_request()
fn_calculate_total_amount()
fn_format_application_status()
```

Boolean を返却する場合は、意味が分かる補助動詞を使用する。

```text
例：
fn_is_admin_user()
fn_has_attachment()
fn_can_submit_request()
```

### 1.13 Boolean 命名

Boolean 値は、以下の接頭語を推奨する。

```text
is_
has_
can_
should_
needs_
requires_
exists_
```

## 2. レイアウト標準

### 2.1 基本方針

画面構成はコンテナベースで設計する。  
固定の `X` / `Y` 座標を直接指定する実装は、原則として避ける。

モバイルアプリの場合、ベースサイズはアプリ設定に従う。  
実機ではスケール表示される前提で、画面内の配置はコンテナを使って調整する。

リキッドレイアウトは初期標準では必須としない。  
まずは、画面全体が端末サイズに応じて拡大・縮小される構成を標準とする。

### 2.2 基本構造

画面は、原則としてルートコンテナを起点に構成する。

```text
root_capture
└─ main_capture
   ├─ header_capture
   ├─ body_capture
   └─ footer_capture
```

画面内の要素は、原則として以下の単位でコンテナ化する。

- ヘッダー
- メインコンテンツ
- 入力エリア
- ボタンエリア
- フッター
- モーダル・確認エリア
- 一覧・カード・明細エリア

### 2.3 固定座標の扱い

`X` / `Y` の直接指定は、原則として使用しない。

位置調整は、以下を優先する。

- コンテナ
- パディング
- ギャップ
- マージン相当の空白コンテナ
- Align / Justify などの配置プロパティ

ただし、コンポーネント内部や装飾目的で厳密な重ね合わせが必要な場合は、限定的に `X` / `Y` 指定を許容する。

### 2.4 垂直・水平・通常コンテナの使い分け

垂直コンテナは、上から下に積む構造に使用する。

```text
例：
header / body / footer
カード内のタイトル / 本文 / ボタン
```

水平コンテナは、左から右に並べる構造に使用する。

```text
例：
戻るボタン / タイトル
アイコン / ラベル
ボタン横並び
```

通常コンテナは、重ね合わせや自由配置が必要な場合に使用する。

```text
例：
画像の上にラベルを重ねる
カードを浮いているように見せる
独自の進捗部品を作る
```

### 2.5 幅・高さ・余白

画面全体で使用する基本幅、高さ、余白は、可能な限り Named Formula または共通定義として管理する。

```powerfx
base_screen_width = 390
base_margin = 16
base_header_height = 56
base_footer_height = 72
base_button_height = 44
```

固定値を各コントロールに直接散らばらせない。  
後から画面サイズ、余白、ボタン高さなどを一括変更できる状態を維持する。

### 2.6 コンテナの伸縮設定

同一階層に配置するコンテナでは、伸縮可能なコンテナは原則1つのみとする。

縦方向コンテナの場合は「高さ（伸縮可能）」、横方向コンテナの場合は「幅（伸縮可能）」を対象とする。

画面全体を `header / body / footer` のように分割する場合は、固定領域と可変領域を明確に分ける。

```text
header_capture：高さ固定
body_capture  ：高さ（伸縮可能）オン
footer_capture：高さ固定
```

同一階層で複数のコンテナを伸縮可能にすると、余白の配分やスクロール位置が読みづらくなり、画面改修時にレイアウト崩れが発生しやすくなる。  
そのため、伸縮可能なコンテナは、原則として「残り領域を受け持つ主領域」に限定する。

例外として、同一階層内で複数領域を意図的に比率配分したい場合のみ、複数の伸縮設定を許容する。

```text
例：
左右2ペインを可変幅で表示する
上下2領域を同じ比率で表示する
複数のカード領域を画面サイズに応じて均等配分する
```

例外として複数の伸縮設定を使用する場合は、各コンテナの役割と期待する表示比率を設計メモまたはコメントで明確にする。

### 2.7 ヘッダー・ボディ・フッター

ヘッダーとフッターは固定高さとし、ボディを可変領域とする。

モバイル画面を `640 x 1136` ベースで設計する場合、目安は以下とする。

```text
header_capture：80〜120
footer_capture：80〜120
body_capture  ：残り領域
```

ボタンやナビゲーションが多い場合は、フッターをやや大きめにする。

### 2.8 「いっぱいに広げる」オプション

幅・高さを明示的に制御するため、`いっぱいに広げる` オプションは原則として使用しない。

理由は以下とする。

- 意図しないサイズ変更が発生しやすい
- 他要素との関係が読み取りにくくなる
- 修正時に影響範囲を予測しづらくなる
- AIや第三者がレイアウト意図を判断しづらくなる

## 3. UI/UX 標準

### 3.1 導線

画面上の導線は、原則として左から右、上から下とする。  
対象ユーザーが日本語・英語など左から右に読む文化圏であることを前提とする。

### 3.2 操作目的と画面構造

利用者が「何をしたいか」と「どこで何ができるか」が紐づきやすいレイアウト・画面遷移とする。

特に以下を意識する。

- 主要操作は視認しやすい位置に置く
- 入力、確認、保存、戻るなどの流れを自然にする
- 同じ意味の操作は、画面間で位置、色、文言をそろえる
- 危険操作は通常操作と視覚的に分離する

### 3.3 操作結果の明示

利用者が操作する要素には、操作後に何が起きるか分かる命名、アイコン、色、アニメーションを採用する。

良い例：

```text
保存
申請する
削除
再読み込み
承認
差戻し
キャンセル
```

曖昧な表現は避ける。

避ける例：

```text
OK
実行
処理
次へ
完了
```

ただし、文脈上明らかな場合は使用してよい。

### 3.4 通知・メッセージ

時間がない場合は `Notify` を使用してよい。  
ただし、重要なエラー、確認、業務判断を伴うメッセージでは、可能な限りインタラクティブな表示を使用する。

標準方針は以下とする。

| 種別 | 表示方法 |
|---|---|
| 軽微な完了通知 | `Notify` |
| 入力エラー | 画面内メッセージ |
| 保存失敗 | 画面内メッセージまたはモーダル |
| 削除・取消などの危険操作 | 確認ダイアログ |
| 業務判断が必要な警告 | モーダルまたは専用エリア |

### 3.5 カラー

色は意味を持たせて使用する。  
装飾目的だけで色を増やさない。

| 色 | 意味 |
|---|---|
| 赤 / 黒 / 紫 | 最重要、深刻、危険、不可逆操作 |
| 黄 / オレンジ | 注意、警告、確認が必要 |
| 青 / グリーン | 通常、成功、進行、肯定的操作 |
| 白 / グレー | ベース、背景、通常表示 |

企業ブランドカラーがある場合は、ブランドカラーを優先する。  
その場合も、意味ごとの色分けは補色、濃淡、アクセントカラーで再設計する。

ナイトモードは将来的な対応候補とし、初期標準では対象外とする。

## 4. カード表現

Power Apps の標準カードにこだわらず、コンテナでカード風の部品を作成してよい。

カード風コンテナでは、以下を設定する。

- 背景色
- 角丸
- 余白
- 必要に応じた影や境界線

```text
例：
application_summary_card
capture_guide_card
result_detail_card
```

カード内は、さらに垂直コンテナや水平コンテナをネストして構成する。

## 5. コンポーネント化方針

複数画面で再利用する部品、または単独で意味を持つ複雑な表示部品は、コンポーネント化を検討する。

```text
例：
進捗ステッパー
共通ヘッダー
共通フッター
検索条件ブロック
エラーメッセージ表示
```

特に、円とバーを組み合わせる進捗表示のように、コンテナだけでは微調整が難しいものは、コンポーネント化した方が保守しやすい。

コンポーネント化する場合は、入力プロパティ、出力プロパティ、責務を明確にする。  
単に見た目だけをまとめるための過剰なコンポーネント化は避ける。

## 6. データ操作標準

### 6.1 Form / Patch の使い分け

フォームで実現できる登録・更新処理は、原則としてフォームを使用する。

`Patch` は以下の場合に限定する。

- 複数レコードを一括更新する
- フォームでは表現しづらい更新制御がある
- 楽観ロックや業務ルールを明示的に制御する
- 更新対象を動的に切り替える必要がある

単純な登録・更新で `Patch` を多用しない。

### 6.2 保存後処理

保存成功後、失敗後、キャンセル時の処理を明確に分ける。

保存後も同じ画面に残る仕様の場合は、保存後に画面上の基準値、選択レコード、Modified 相当の比較値などを更新する。  
古い比較値を保持したまま再保存しない。

### 6.3 エラーハンドリング

データ操作では、利用者にとって意味のあるエラーメッセージを表示する。

技術的なエラー内容をそのまま表示するのではなく、利用者が次に何をすればよいか分かる表現にする。

## 7. 状態管理標準

### 7.1 `App.OnStart` に多くの処理を詰め込まない

`App.OnStart` に初期化処理を詰め込みすぎない。

現在の Power Apps では、非ブロッキング `App.OnStart` が標準的な動作である。  
また、非ブロッキング動作を無効化する設定は廃止済みに分類されているため、当該設定に依存した設計は行わない。

そのため、`App.OnStart` で初期化した変数・コレクションが、スタート画面の `OnVisible` や各コントロールから即時参照される前提の実装は避ける。

理由は以下とする。

- `App.OnStart` と画面側処理の実行順序が保証されない場合がある
- 非ブロッキング `App.OnStart` が有効な場合、`App.OnStart` と `Screen.OnVisible` は並行して実行され得る
- `App.OnStart` で初期化した変数・コレクションを参照すると、初期化前の値を参照するリスクがある
- 初期化処理の依存関係が見えにくくなり、保守時の原因調査が難しくなる

`App.OnStart` は、アプリ全体で一度だけ必要な軽量な初期化に限定する。  
画面表示に必要な値は、画面単位の `OnVisible`、Named Formula、または必要箇所での遅延評価に寄せる。

### 7.2 変数の使い分け

グローバル変数は、アプリ全体で共有する状態に限定する。

ローカル変数は、画面内だけで完結する一時状態に使用する。

Named Formula は、読み取り専用の計算値、参照値、共通定数に使用する。

処理の再利用には、User Defined Function を使用する。

## 8. Power Fx 実装標準

### 8.1 `With` の過剰ネスト

`With` のネストは原則として避ける。

- 1層目：許容
- 2層目：非推奨
- 3層目以上：強く非推奨

`With` をネストする前に、以下を検討する。

- `As` 句で読みやすくできないか
- Named Formula に分離できないか
- 変数またはコレクションに分けた方がよいか
- User Defined Function に切り出せないか
- 処理をコンポーネント側に寄せられないか

### 8.2 `If` の過剰ネスト

`If` のネストは原則として避ける。

- 1層目：許容
- 2層目：非推奨
- 3層目以上：強く非推奨

複数条件の分岐は、可能な限り `Switch`、条件判定用の Named Formula、または User Defined Function に分離する。

### 8.3 `!IsBlank()` の乱用

単純な代替値の指定には、`!IsBlank()` ではなく `Coalesce()` を優先する。

```powerfx
Coalesce(input_value, default_value)
```

ただし、空白かどうか自体を判定したい場合は `IsBlank()` を使用してよい。

### 8.4 否定条件の乱用

条件式で `!A_status` や `Status <> "A"` のような否定条件を安易に使用しない。

複数ステータスが存在する場合は、原則として対象ステータスを列挙する。

```powerfx
Status in ["B", "C"]
```

理由は以下とする。

- 将来 `D` が追加された場合に、意図せず対象に含まれるリスクがある
- 読み手が全ステータスを完全に把握しているとは限らない
- 条件の業務意図が伝わりにくい
- コンボボックスの `Default` / `DefaultSelectedItems` 系プロパティで使用した場合、参照先データや初期値の評価タイミングによって、想定どおり選択状態が復元されないことがある

否定条件は、全体集合が明確で、将来追加される値も含めて意図的に「A以外」としたい場合のみ使用する。

### 8.5 コピペ実装

同じような処理を3回コピーする前に、関数設計、部品化、共通化を検討する。

検討順序は以下とする。

1. User Defined Function で共通化できないか
2. Named Formula で読み取り専用値として共通化できないか
3. コンポーネント化できないか
4. コレクションや設定テーブルで制御できないか
5. 画面構造や処理配置を見直せないか

理想は、実装前に以下を設計しておくことである。

- User Defined Function
- Named Formula
- コンポーネント
- 画面ごとの責務
- 機能ごとの処理配置

---

## 9. 非推奨事項 / Don’ts

以下は、保守性、可読性、実行順序、将来変更への耐性を下げやすいため、原則として避ける。

| 非推奨 | 代替方針 |
|---|---|
| `App.OnStart` に多くの処理を詰め込む | 画面単位の `OnVisible`、Named Formula、遅延評価に寄せる |
| 単純更新で `Patch` を多用する | Form / SubmitForm を優先する |
| `With` を深くネストする | `As`、Named Formula、User Defined Function に分離する |
| `If` を深くネストする | `Switch`、判定式、User Defined Function に分離する |
| `!IsBlank()` で代替値を制御する | `Coalesce()` を使用する |
| `Status <> "A"` のような否定条件を多用する | 対象値を列挙する |
| 同じ処理を何度もコピペする | 3回目の前に共通化する |
| 固定の `X` / `Y` に依存する | コンテナ、余白、配置プロパティを使う |
| 画面ごとに色や文言を変える | 意味ごとの色、文言、位置を統一する |
| Named Formula を関数のように扱う | User Defined Function を使用する |

## 10. 実装時の判断基準

迷った場合は、以下の順で判断する。

1. 後から見て役割が分かるか
2. 画面構造をツリーで見たときに理解しやすいか
3. 固定座標に依存しすぎていないか
4. 変更時に影響範囲が局所化されているか
5. コントロール名がアプリ全体で一意か
6. 命名が長すぎず、短すぎないか
7. AIや第三者が意図を誤解しにくいか

本標準は、過剰な抽象化よりも、実装者が迷わず作れることを優先する。  
迷った場合は、短く書くことよりも、誤解されにくく書くことを優先する。
