# Power Apps キャンバスアプリ開発標準

作成日：2026-06-18

## 1. 基本方針

本標準は、Power Apps キャンバスアプリを、保守しやすく、読みやすく、実装者間で迷いにくい形で作成するための開発標準である。

前提は以下とする。

- モダンコントロールを主に使用する
- SharePoint リストを主要データソースとして使用する
- 固定の `X` / `Y` 指定は極力使用しない
- コンテナベースでレイアウトを構成する
- レスポンシブ対応は必須としない
- 命名はスネークケースを基本とする

## 2. 命名規則

### 2.1 基本ルール

コントロール名、変数名、関数名、データソース名は、原則としてスネークケースで命名する。

```text
例：
search_button
capture_result_screen
body_container
sp_license_application
```

命名では、以下を意識する。

- 何の部品か分かること
- どの役割か分かること
- 長すぎないこと
- 画面内で見たときに探しやすいこと

### 2.2 コントロール名

コントロール名は、役割を先に置き、種類は必要に応じて末尾に付ける。

```text
例：
back_button
search_button
submit_button
license_image
progress_bar
progress_circle
```

画面単位で一意であればよい。アプリ全体で無理に完全一意を目指さない。

ただし、共通部品や複数画面で同名になりやすいものは、必要に応じて画面・用途を補足する。

```text
例：
back_to_result_button
back_to_search_button
```

### 2.3 画面名

画面名は、画面の業務上の役割を表す。

```text
例：
home_screen
capture_screen
capture_result_screen
search_result_screen
```

### 2.4 コンテナ名

コンテナ名は、見た目ではなくレイアウト上の役割で命名する。

```text
例：
root_container
main_vertical_container
header_container
body_container
footer_container
content_container
section_container
card_container
```

コンテナの方向を明確にしたい場合は、必要に応じて `vertical` / `horizontal` を含める。

```text
例：
main_vertical_container
action_horizontal_container
```

ただし、すべてのコンテナに方向名を入れる必要はない。役割名だけで十分に分かる場合は、簡潔さを優先する。

### 2.5 データソース名

SharePoint リストは、接頭辞 `sp_` を付ける。

```text
例：
sp_license_application
sp_vehicle_usage_request
sp_admin_user
```

Dataverse テーブルや他データソースを使用する場合も、必要に応じて接頭辞で判別できるようにする。

```text
例：
dv_account
dv_contact
```

### 2.6 変数名

グローバル変数は `glb_`、コンテキスト変数は `loc_` を接頭辞とする。

```text
例：
glb_login_user
glb_selected_application
loc_is_loading
loc_error_message
```

### 2.7 コレクション名

コレクションは `col_` を接頭辞とする。

```text
例：
col_search_results
col_application_items
```

### 2.8 名前付き式

名前付き式は、読み取り専用の計算値・参照値として使用する。

命名は、値の意味が分かる名詞または名詞句とする。

```text
例：
login_user
current_user_mail
is_admin_user
base_screen_width
base_screen_height
```

名前付き式をグローバル変数経由で関数のように使うことはしない。

### 2.9 ユーザー定義関数

処理を再利用する場合は、ユーザー定義関数の利用を強く推奨する。

命名は、処理内容が分かる動詞句とする。

```text
例：
get_login_user()
is_admin_user()
format_application_status()
```

## 3. レイアウト標準

### 3.1 基本構造

画面は、原則としてルートコンテナを起点に構成する。

```text
root_container
└─ main_vertical_container
   ├─ header_container
   ├─ body_container
   └─ footer_container
```

モバイルアプリの場合、ベースサイズはアプリ設定に従う。実機ではスケール表示される前提で、画面内の配置はコンテナを使って調整する。

### 3.2 固定座標の扱い

`X` / `Y` の直接指定は、原則として使用しない。

位置調整は、以下を優先する。

- コンテナ
- パディング
- ギャップ
- マージン相当の空白コンテナ
- Align / Justify などの配置プロパティ

ただし、コンポーネント内部や装飾目的で厳密な重ね合わせが必要な場合は、限定的に `X` / `Y` 指定を許容する。

### 3.3 垂直・水平・通常コンテナの使い分け

垂直コンテナは、上から下に積む構造に使用する。

```text
例：
header / body / footer
カード内のタイトル / 本文 / ボタン
```

水平コンテナは、左から右に並べる構造に使用する。

```text
例：
戻るボタン / タイトル
アイコン / ラベル
ボタン横並び
```

通常コンテナは、重ね合わせや自由配置が必要な場合に使用する。

```text
例：
画像の上にラベルを重ねる
カードを浮いているように見せる
独自の進捗部品を作る
```

### 3.4 コンテナの伸縮設定

同一階層に配置するコンテナでは、伸縮可能なコンテナは原則1つのみとする。

縦方向コンテナの場合は「高さ（伸縮可能）」、横方向コンテナの場合は「幅（伸縮可能）」を対象とする。

#### 基本方針

画面全体を `header / body / footer` のように分割する場合は、固定領域と可変領域を明確に分ける。

```text
header_container：高さ固定
body_container  ：高さ（伸縮可能）オン
footer_container：高さ固定
```

この構成により、ヘッダー・フッターの高さを固定し、残りの表示領域をボディが自然に埋めることができる。

#### 理由

同一階層で複数のコンテナを伸縮可能にすると、余白の配分やスクロール位置が読みづらくなり、画面改修時にレイアウト崩れが発生しやすくなる。

そのため、伸縮可能なコンテナは、原則として「残り領域を受け持つ主領域」に限定する。

#### 例外

同一階層内で複数領域を意図的に比率配分したい場合のみ、複数の伸縮設定を許容する。

```text
例：
左右2ペインを可変幅で表示する
上下2領域を同じ比率で表示する
複数のカード領域を画面サイズに応じて均等配分する
```

ただし、例外として複数の伸縮設定を使用する場合は、各コンテナの役割と期待する表示比率を設計メモまたはコメントで明確にする。

### 3.5 ヘッダー・ボディ・フッター

ヘッダーとフッターは固定高さとし、ボディを可変領域とする。

モバイル画面を `640 x 1136` ベースで設計する場合、目安は以下とする。

```text
header_container：80〜120
footer_container：80〜120
body_container  ：残り領域
```

ボタンやナビゲーションが多い場合は、フッターをやや大きめにする。

## 4. カード表現

Power Apps の標準カードにこだわらず、コンテナでカード風の部品を作成してよい。

カード風コンテナでは、以下を設定する。

- 背景色
- 角丸
- 余白
- 必要に応じた影や境界線

```text
例：
application_summary_card_container
capture_guide_card_container
result_detail_card_container
```

カード内は、さらに垂直コンテナや水平コンテナをネストして構成する。

## 5. コンポーネント化方針

複数画面で再利用する部品、または単独で意味を持つ複雑な表示部品は、コンポーネント化を検討する。

```text
例：
進捗ステッパー
共通ヘッダー
共通フッター
検索条件ブロック
エラーメッセージ表示
```

特に、円とバーを組み合わせる進捗表示のように、コンテナだけでは微調整が難しいものは、コンポーネント化した方が保守しやすい。

## 6. 実装時の判断基準

迷った場合は、以下の順で判断する。

1. 後から見て役割が分かるか
2. 画面構造をツリーで見たときに理解しやすいか
3. 固定座標に依存しすぎていないか
4. 変更時に影響範囲が局所化されているか
5. 命名が長すぎず、短すぎないか

本標準は、過剰な抽象化よりも、実装者が迷わず作れることを優先する。
